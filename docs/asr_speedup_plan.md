# podedit ASR 改善計画書

文字起こし(faster-whisper)の **速度** と **精度** を上げるための実装計画。
編集部の業務(プレジデントオンライン編集者+同僚数名が `feat/podedit-recovery` の Codespace を共有)向け。

---

## 結論(忙しい人向け)

**まず最初に効くのはこの3つ。それ以外は後回しでよい。**

1. **計測基盤** — 評価セット(代表音源 2 本 + 編集者修正済み正解)と「編集者の文字修正クリック数」KPI を整える。以後のすべての施策はこの数値で採否を判定する。
2. **辞書置換(post-processing)** — 「黒だ → クロード」のようなパターンを編集者が登録し、ASR 出力に自動適用。**モデルを触らずに当日の業務に効く。**
3. **GPU の効果実測** — Codespace GPU machine で `large-v3-turbo` を1日ベンチして、コストと精度向上の比率を見る。本実装はこの数字を見てから判断する。

「30分音源を 2-5分」は GPU が必要、「誤認識の体感半減」は辞書 + glossary でほぼ達成可能、というのが現状の見立て。

---

## 1. 現状ベースライン (2026-05-14 計測)

| 環境 | モデル | beam | wall (30min 音源) | RTF | 主観精度 |
|---|---|---|---|---|---|
| Codespace CPU / int8 | tiny | 1 | ~3.4 min | 0.11 | 「黒だ」等 多い |
| Codespace CPU / int8 | small | 1 | ~12 min | 0.41 | 固有名詞は当たり始める |
| Codespace CPU / int8 | small | 5 | ~17 min | 0.56 | quality preset |

- 16kHz / mono に ffmpeg 変換後、faster-whisper に渡している
- 既に prompt biasing (`日本語のポッドキャスト会話。…`) は適用済み
- model handle のキャッシュ済み(再起動しない限り 3-10s の load を再支払いしない)

**目標**: 30分音源を **2-5分** で文字起こし、固有名詞の誤認識を **体感半減**。

---

## 2. ボトルネック診断

- 速度: CPU + int8 のままでは小幅な改善(2-4x)が限界。**目標値 (2-5min) には GPU が事実上必須。**
- 精度: モデルを `small` から `large-v3-turbo` に上げれば日本語の固有名詞は大幅改善するが、CPU では遅すぎて運用不可。
- 業務インパクト: 編集者は **「クロード」「スタンフォード」「認知バイアス」**等の固有名詞修正に時間を取られている。**モデル精度より glossary / 辞書置換の方が費用対効果が高い場合あり。**

---

## 3. 改善項目一覧と優先順位

各項目: 概要 / 期待効果 / コスト (S=1日以内, M=半日〜3日, L=1週以上) / 副作用。

### P0-0(これがないと他の P0 を評価できない)

0. **計測基盤の整備** — コスト: **M**
   - 代表音源 2 本 + 編集者が確認した正解 transcript
   - 速度指標: 既存 `benchmarks.jsonl` の RTF + 「Transcribe ボタンを押してから編集着手まで」の wall
   - 精度指標(主): **編集者の文字修正クリック数 / 1時間音源** を KPI 化、UI に既存の op カウンタを再利用
   - 精度指標(補): 固有名詞ヒット率を案件辞書で別途集計
   - **以後の全施策はこの数値で採否を決める** ことを明文化

### P0(目標達成の本筋)

1. **I. Post-processing 辞書置換** — コスト: **S-M**
   - 編集者が「ASR の出力 → 正しい綴り」のペアを登録(例: `黒だ → クロード`)
   - transcript 生成直後にサーバ側で一括置換、`note: "dict:..."` で記録
   - 効果: モデル変更不要で「いつもの誤認識」を恒久修正
   - 副作用: 過剰置換のリスク → 編集者がワンクリックで undo できる UI が必要

2. **H. Glossary 永続化** — コスト: **M**
   - 編集者が固有名詞リスト (1行1語) を UI から登録 → `work_dir/glossary.txt`
   - 文字起こし時に `initial_prompt` に自動結合 + `hotwords` パラメータに渡す
   - 効果: 「クロード」「Anthropic」など案件特有の語の誤認識を半減
   - 副作用: prompt 長すぎると逆効果、~1000 字で切り詰める

3. **A-prep. GPU ベンチマーク(本実装ではなく検証)** — コスト: **S**
   - Codespace GPU machine を 1〜2 時間借りて、`large-v3-turbo` / CUDA / float16 を計測
   - 30 分音源で wall・RTF・固有名詞ヒット率・課金額を実測
   - **このベンチ結果を見てから本実装(A 本体)に投資するか判断**

4. **N. 進捗 RTF 表示** — コスト: **S**
   - 既に `/api/library/transcribe/status` で進捗を返している → UI に「現在 RTF 0.42, 残り推定 3分」を表示
   - 効果: 待ち時間の体感 -30〜50%
   - 副作用: なし

### P1(P0 検証後、計測値次第で順位入替えあり)

5. **A. GPU 実行対応(本実装)** — コスト: **M-L**
   - A-prep の結果が良ければ着手
   - `device="cuda"` + `compute_type="float16"` で動かす、README に GPU machine 切替手順
   - 効果: small/CPU → large-v3-turbo/CUDA で 10-15x、精度も同時に上がる

6. **G. デフォルト model を large-v3-turbo に**(GPU 利用前提) — コスト: **S**
   - A 完了直後ではなく、**評価セットで「速度・修正クリック数・費用」が確認できてから** 切替
   - 効果: 固有名詞精度 +20-30%

7. **O. 部分結果の順次表示** — コスト: **M**
   - faster-whisper は segment ごとに yield → 完了済み segment を順次 transcript.json に追記
   - UI は polling で増分取得 → 完了を待たず先頭から編集着手可
   - 副作用: 編集中に transcript が伸びる UX、scrolling 制御要 → **事故りやすいので分離して着手**

8. **M. プリセット文言を業務寄りに再整理** — コスト: **S**
   - `fast/balanced/quality` → 「下書き最速 / 通常編集 / 固有名詞重視」など編集者用語に

9. **P. 「ここから再認識」ボタン** — コスト: **M**
   - 範囲選択 + 「再認識」ボタンで該当区間だけ別モデル(`large-v3-turbo`)で再実行
   - transcript はサブ区間だけ置換

10. **C. VAD で沈黙 skip** — コスト: **M**
    - 評価セットで `min_silence_duration_ms` を 300/500/700 でスイープ
    - 沈黙の多い対談ポッドキャストで 1.5x

11. **B. 音源並列分割** — コスト: **M-L**
    - 30分音源を 5分×6 に切って並列処理、境界 10-20秒 overlap + timestamp merge
    - 効果: CPU でも 4-6x

### P2(将来検討)

12. **J. 信頼度低区間だけ 2-pass 再認識** — コスト: **L**(`avg_logprob` 閾値設計に評価セット必須)
13. **D. モデルプリロード強化** — コスト: **S**(初回 3-10s 短縮のみ)
14. **E. faster-whisper batched API** — コスト: **M**(GPU 前提、CPU では効果薄)
15. **F. 派生モデル (kotoba-whisper, distil-whisper)** — コスト: **M**(評価セット必須)
16. **K. `condition_on_previous_text=True`** — コスト: **S**(並列分割と相性悪い、要検証)
17. **L. VAD パラメータの細部チューニング** — コスト: **S-M**

## 5. リスクと未解決事項

- **Codespace GPU の利用可否と課金**: 時間単価が高い。月額予算と「GPU を使う/使わない」のスイッチ UI、人数共有時の競合(GPU 1 枚を 4 人で奪い合うのか?)
- **CUDA / 環境差分**: ローカル macOS の人は Metal が必要、Windows + WSL は別パス。サポート範囲を絞る。
- **音源・transcript の保存先と権限**: 共有 Codespace で各人の音源が混じる可能性。`work_dir/<person>/...` のように分けるか?
- **辞書の誤置換**: 「会う → クロードに会う」のような過剰置換。1クリック undo を必須に。
- **評価セット作成の編集者負担**: 「正解 transcript を作る」のは数時間かかる。最初は 1 本だけでも始める。
- **モデル / ライブラリ更新の再現性**: faster-whisper / CTranslate2 のバージョン pin を `uv.lock` でしっかり管理。
